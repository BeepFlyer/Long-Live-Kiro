# 任务索引库 — 创建与维护规则

本文档包含创建和维护 task 文件的详细规则。日常读取索引和触发加载的规则已在 POWER.md 中，本文档仅在需要创建新 task 或维护现有 task 时通过 `readSteering` 读取。

## 为什么拆分

- 并行任务多时，详情直接写在 `current.md` 会膨胀灵性记忆，挤占上下文空间
- 索引+详情分离后，基础开销只有索引行（每条约 20 字），即使 10 个并行任务也只占约 200 字符
- 详情按需加载，不用的任务零开销

## 单个 task 文件格式

```markdown
# <话题名>

## 状态
<emoji> <状态> | 创建: 第N世 | 最后更新: 第M世

## 标签
`tag1`, `tag2`, `tag3`

## 上下文
（核心背景，让下一世能快速理解这件事是什么、为什么做）

## 当前进度
（最新状态，断点在哪）

## 待办
- [ ] 下一步 A
- [ ] 下一步 B

## 相关技能
（首次加载时由标签匹配自动填充，也可手动添加）
- skills/example/some-skill.md

## 关键决策
（如有重要的选型/取舍记录）
```

## 标签选取规则

- 每个任务 2-5 个标签，用反引号包裹
- 标签来源：领域词（`Unity`, `Lua`, `记忆系统`）、功能词（`bugfix`, `架构`, `UI`）、实体词（`居民`, `瞭望塔`, `POWER`）
- 标签要具体到能区分不同任务，避免过于宽泛（如单独一个 `代码` 没有区分度）
- 新建任务时必须设标签，蒸馏时可微调

## 技能匹配

加载 task 文件后，取其标签与 `skills/index.md` 的标签做交叉匹配。

- 命中的技能路径追加到 task 文件的"相关技能"字段中（首次匹配时写入）
- 后续加载该 task 时直接看"相关技能"字段，无需重复匹配
- 手动添加的技能引用同样有效，不限于自动匹配

这使得技能召回从"AI 自觉查询"变为"task 驱动自动关联"，降低遗忘风险。

## 篇幅限制

单个 task 文件上限：**800 中文字符**（约 400-500 tokens）。

- 读一个 task 文件的开销必须可控，不能吃掉大量上下文
- 800 字符足够承载"上下文 + 进度 + 待办"的核心信息
- 超过 800 字符时：压缩已完成的待办，提炼稳定上下文为一句话结论，或拆分为多个子任务
- 复杂任务的稳定背景知识应毕业到技能库，task 文件只保留动态进度

## 生命周期

| 阶段 | 操作 |
|------|------|
| 创建 | 新任务出现时，在 `tasks/` 下创建文件，在索引中添加一行 |
| 更新 | 任务有进展时，更新 task 文件内容 + 索引中的状态 emoji |
| 完成 | 从索引中删除该行，task 文件保留 7 世后删除（或用户手动清理） |
| 归档 | 完成的任务如有长期价值（经验/决策），提炼后写入灵性记忆的"经验"或"决策日志"，然后删除 task 文件 |

## 与其他记忆层的关系

- **灵性记忆**：`current.md` 的"进行中"栏目保留轻量索引，详情在 task 文件中
- **技能库**：tasks 是"进行中的工作"，skills 是"沉淀的知识"，定位不同
- **硬性记忆**：不受影响
- **备份**：`tasks/` 不单独备份，但备份文件中包含任务索引快照（current.md 的"进行中"栏目），足以恢复方向
